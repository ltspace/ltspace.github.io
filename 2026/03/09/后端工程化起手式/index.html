后端工程化起手式

发表于 2026-03-09 分类于 教程

很多后端项目不是死在复杂业务上，而是死在一开始就没把基本盘搭好。

刚开项目时，大家通常都很急：先把接口跑起来，先把表建出来，先把功能交付掉。结果三周后开始补格式化，补迁移，补测试，补统一响应，补分页，补时间字段，补错误码，补日志，最后项目越补越乱。

我现在更倾向于把这些事情前置。因为绝大多数“后面再说”，最后都会变成技术债，而且是那种每天都在磨人的技术债。

这篇就记录一下，我现在会怎么起一个后端项目，尤其是 FastAPI 项目。

先把项目骨架搭起来

一个新项目开始时，最先做的不是写业务接口，而是把工程边界确定下来。

如果项目已经确定要用 isort 和 black，那就不要拖到后面再装。格式化工具越晚接入，历史代码越难收敛。

初始化之后，我会立刻把最基础的工程约束加上：

  • 目录结构先定下来，不要等代码长出来以后再搬家
  • 数据库迁移先接上，不要靠手改表结构
  • 代码格式化和提交检查先接上，不要把风格问题留给人工 review
  • 本地开发环境和测试环境的数据库命名、连接方式尽量一致

项目小的时候，这些动作看起来像“讲究”；项目一旦开始迭代，它们就是最基本的秩序。

提交前的自动检查要尽早接入

我基本不会再接受“等要发版了再整理代码风格”这种做法。

最省事的方式就是直接把 pre-commit 接进来，至少做三件事：

  • 检查 YAML
  • 去掉行尾空格和文件末尾缺失换行
  • 统一 import 顺序和代码格式

Model 要尽早补全，不要边写边凑

我现在很强烈的一个习惯是：新项目起来之后，先把核心 model 想清楚。

因为 model 不是“数据库那一层的事情”，它会反过来影响：

  • 表结构怎么设计
  • 接口输入输出长什么样
  • 分页、筛选、排序怎么做
  • 数据校验放在哪里
  • 迁移怎么管理

如果 model 从一开始就是半成品，后面所有层都会跟着松。

这部分我会优先确定几件事：

  • 主键类型是不是统一用 BIGINT
  • 命名是不是统一用蛇形
  • created_at、updated_at 这些时间字段是不是全局统一
  • 业务时间和系统时间是不是分清楚
  • 分页参数是不是一开始就有统一组件，比如 page_num 和 page_size

很多人会把这些留给“后面抽公共模块的时候再做”。但经验上，真正该统一的东西越晚统一，代价越高。

接口规范不要靠默契

后端最容易烂掉的地方之一，就是“看着都能跑，但每个接口都长得不一样”。

我现在会在项目一开始就尽量收口这几件事：

  • API 路径命名统一
  • 响应结构统一
  • 错误提示风格统一
  • 分页返回统一

比如路径层面，我更倾向于尽早定清楚规范，不要一会儿短横线，一会儿下划线，一会儿单数，一会儿复数。规则本身不是最关键的，关键是整个项目要一致。

时间、迁移、索引这些基础设施别拖

很多后端问题不是功能写错，而是基础设施没立起来。

至少有三件事，我现在会在项目早期就接好：

1. 迁移历史要走 Alembic

不要靠手工改表，也不要把“数据库结构变更”藏在 README 或群消息里。表结构变化必须有迁移记录。

2. 时间处理要统一

时间字段是最容易悄悄把系统弄乱的东西。

我会尽量避免这几种情况：

  • 一部分字段存本地时间，一部分存 UTC
  • 一部分接口返回字符串，一部分返回时间戳
  • 项目里继续使用容易混淆或已不推荐的时间 API

时间语义一旦不统一，排查问题会非常难。

3. 索引和查询代价要有意识

项目刚开始时数据量小，任何 SQL 看起来都不慢。但只要接口成型，下面这些问题通常都会冒出来：

  • N+1 查询
  • 无索引排序
  • 深分页
  • 全表扫

所以我倾向于尽早关注关键查询，至少知道哪些字段未来大概率需要索引，而不是等线上慢了再补。

测试不是最后补的，是一开始就接入的

我越来越不相信“先写完功能，后面再补测试”。

现实里，“后面”通常不会来。

至少对于核心逻辑，我会尽量早一点把下面两类测试接进去：

  • 单元测试
  • 集成测试

更进一步的话，最好让测试和提交动作挂钩。哪怕不是每次都跑全量，也应该让关键校验自动执行。

这件事的收益并不只是“防回归”，还有一个很实际的价值：它会倒逼你把代码写得更可测试。很多结构糟糕的问题，一到写测试的时候就暴露出来了。

写接口之前，先做一轮自检

项目能跑，不代表项目健康。

我现在会用一份很朴素的自检清单反复看项目，尤其在接口开始变多之后。关注点通常包括：

  • 关键链路最慢的步骤是什么
  • 外部依赖有没有超时、重试和上限
  • 连接池和并发模型是不是合理
  • 写操作有没有幂等约束
  • 上传和对象存储有没有唯一命名策略
  • 入参校验、大小限制和安全边界是否明确
  • 密码、密钥、token 是否只从安全配置读取
  • 生命周期钩子和框架用法有没有踩到弃用能力
  • 日志、trace、指标和错误码能不能支撑排查
  • 限流、熔断、降级、回滚是否有基本预案

这类问题不用等到“系统很大了”再考虑。越早有意识，后面越省力。

一个项目开始时最容易忽略的事

如果要把上面的内容再压缩一下，我觉得最容易被低估的是这几件事：

  1. 不要把临时方案当默认方案
  2. 不要等代码变多了再统一风格
  3. 不要等表结构长歪了再补迁移
  4. 不要等接口满天飞了再定响应规范
  5. 不要等线上出问题了才开始补日志、指标和测试

后端工程化听起来像“大厂词汇”，但实际落下来并不神秘。无非就是把那些迟早都要补的基础动作，提前做掉。

很多项目最后并不是输在难点，而是输在一开始对混乱过于宽容。

对我来说，“起手式”的核心其实就一句话：

先把秩序搭起来，再去堆功能。
